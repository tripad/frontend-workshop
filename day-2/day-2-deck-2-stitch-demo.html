We're about to design a two-screen ride hailing prototype using Stitch, Google's AI design tool, driven from Claude Code via MCP.

This demo is doing two things at once.

One: showing you Stitch. It's a different shape of tool from what you saw yesterday. Stitch generates designs, not code. Useful when you want to think in screens.

Two: showing you how MCP servers let your AI agent control other tools. Yesterday, Antigravity wrote code for you. Today, the same kind of agent will drive Stitch through an MCP connection.

After we generate the screens, we come back to states.

Quick look at what Stitch gave us.

Both screens are competent. They show what happens when everything works: pickup is valid, fare loads, driver is assigned, ETA is reasonable.

Neither shows what happens when things go differently. That's not Stitch's fault. It's what we asked for. We described the happy path. We got the happy path.

- Empty states

An activity for your trio project, not for right now

Find your missing states

You're building your trio project as a real working prototype: a few HTML screens, maybe a dummy API or two.

Open your Figma file alongside your code. Every time you hit a state in code that wasn't in your Figma file, add it to a list.

Loading screens you didn't draw. Empty states you didn't think of. Errors the API can throw. Edge cases that only show up when you wire things together.

This is the gap between what you designed and what a real product needs. The act of building will surface it.

Bring the list to Day 3.

Missing states from my food delivery prototype:

  • cart loading
  • address invalid
  • no delivery to pincode
  • payment timeout
  • restaurant closed
  • coupon code expired
